# NetBird Dashboard i18n/Han 化交接文档

## 1. 当前 i18n 架构

项目已经接入了一个轻量级的内置 i18n 方案，核心文件如下：

- `src/i18n/messages.ts`
  维护所有文案字典。
  当前已存在两个 locale：
  `en` 和 `zh-CN`。
  当前默认语言已经被设置为：
  `zh-CN`。

- `src/i18n/I18nProvider.tsx`
  提供 `useI18n()` Hook。
  主要能力：
  `locale`、`setLocale()`、`t(key, values?)`。
  同时会在 locale 变化时同步：
  `document.documentElement.lang`
  和
  `dayjs.locale(...)`。

当前实现方式不是依赖 `next-intl` / `react-intl`，而是项目内自维护字典，所以继续汉化时应沿用现有模式，不要混入新的 i18n 框架。

## 2. 当前约定

### 2.1 文案接入方式

在 React 组件里优先使用：

```tsx
import { useI18n } from "@/i18n/I18nProvider";

const { t } = useI18n();

<Button>{t("actions.save")}</Button>
```

### 2.2 文案 key 命名方式

项目目前已经形成较稳定的命名习惯：

- `common.*`
- `actions.*`
- `nav.*`
- `table.*`
- `filters.*`
- `模块名.*`

例如：

- `common.cancel`
- `actions.saveChanges`
- `nav.controlCenter`
- `reverseProxy.targetAdded`
- `userDetails.lastLoginFormat`

继续补文案时，尽量复用已有 key，不要重复造新的近义 key。

### 2.3 日期/时间

项目已经在 `I18nProvider` 中设置了 `dayjs.locale(...)`。
但是很多页面以前直接写死了：

```tsx
dayjs(value).format("D MMMM, YYYY [at] h:mm A")
```

这会导致中文环境里仍然出现英文 `at`。

当前推荐做法是把格式串也放进 messages：

```tsx
dayjs(value).format(t("userDetails.lastLoginFormat"))
```

目前已经新增并使用的格式 key：

- `userDetails.lastLoginFormat`
- `peerDetails.dateTimeFormat`

后续如果别的页面也需要长日期格式，建议统一抽为更通用的 key，例如：
`dateTime.long`
或
`common.dateTimeLong`
但暂时项目内还没有彻底统一。

### 2.4 Metadata 标题约定

`src/app/**/layout.tsx` 里的 `metadata.title` 目前不跟随前端语言切换。

原因是当前 i18n 状态主要来自客户端 `localStorage`，而 Next 的 `metadata` 定义属于静态/服务端层，拿不到客户端当前语言。

因此当前项目约定是：

- metadata 标题保持英文原文
- 不把 `layout.tsx` 的标题硬编码成中文
- 真正的 metadata 多语言要等后续引入服务端可感知的 locale 方案后再统一处理

## 9. 汉化时的具体步骤

建议每处理一个文件都遵循同样流程：

1. 先读文件，识别“用户能看到的文案”。
2. 优先复用已有 key。
3. 如果没有合适 key，再往 `src/i18n/messages.ts` 里补一对 `en/zh-CN` 文案。
4. 在组件里接入 `useI18n()`。
5. 替换硬编码文本。
6. 检查日期格式、空状态、按钮、tooltip、placeholder、表头、弹窗标题。
7. 检查是否误改了内部值、路由参数、枚举字符串。
